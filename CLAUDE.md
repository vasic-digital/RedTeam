# CLAUDE.md -- digital.vasic.redteam

Module-specific guidance for Claude Code.

## Purpose

`digital.vasic.redteam` is a YAML-driven adversarial prompt fixture
harness for defensive LLM guardrail regression testing. It loads
attack-class fixtures from an embedded YAML corpus and hands them to
consumers, which replay them through their own guardrail pipeline and
assert every fixture is blocked.

Pure stdlib + `gopkg.in/yaml.v3`; no network I/O; no live model calls.

## Hard rules (non-negotiable)

1. **NO CI/CD pipelines** -- no `.github/workflows/`, `.gitlab-ci.yml`,
   `Jenkinsfile`, `.travis.yml`, `.circleci/`, or any automated
   pipeline. No Git hooks either. Permanent.
2. **SSH-only for Git** -- `git@github.com:...` / `git@gitlab.com:...`.
   Never HTTPS, even for public clones.
3. **Conventional Commits** -- `feat(redteam): ...`, `fix(...)`,
   `docs(...)`, `test(...)`, `refactor(...)`. Every commit generated by
   an agent ends with a `Co-Authored-By:` trailer.
4. **Code style** -- `gofmt`, `goimports`, 100-char line ceiling
   (readability first), errors always checked and wrapped with
   `fmt.Errorf("...: %w", err)`.
5. **Tests with resource cap** --
   `GOMAXPROCS=2 nice -n 19 ionice -c 3 go test -count=1 -p 1 -race ./...`

## Dependencies

- `gopkg.in/yaml.v3` (fixture parsing)
- `github.com/stretchr/testify` (tests only)

## Primary consumer

HelixAgent (`dev.helix.agent`) -- consumed via `go.mod` replace
directive pointing at the `RedTeam/` submodule. The in-tree consumer is
`internal/security/redteam_fixtures.go` (`DeepTeamRedTeamer.RunFixtureSuite`).

## Defensive use

This package is explicitly defensive. It parses and returns adversarial
fixtures so that consumers can verify their own guardrail pipelines
block every known attack pattern. See `AGENTS.md` for policy
enforcement when touching this module.
